# Frontend-Accelerator-Pre-Corse-5th 🔥

## 개요
구체적인 설계를 잡고 코드를 작성하기 보다 점진적 리팩토링을 통해 가장 큰 단위로 나눌 수 있는 코드를 분리하고 그 안에서 모듈화를 진행하였습니다.

`SavingsCalculatorPage.tsx`에서 페이지의 구조가 한 눈에 보이게 Form 과 Tab 컴포넌트로 분리하고 가장 상단에서 필요한 상태값을 props로 전달하여 사용하게 하였습니다.

## 내가 생각하는 추상화와 확장 가능성
- 추상화: 기본적으로 "무엇을 숨기고 무엇을 드러낼 것인가"를 의미한다고 생각합니다. 이 코드를 사용하는 입장에서 꼭 알아야할 정보는 무엇인지 생각하며 알 필요가 없는 코드는 안으로 숨기고 알아야만 하는것은 인터페이스로 노출하는 형태로 구현합니다.
    - 현재 코드에서 추상화에 신경 쓴 부분
        1. 커스텀 훅
            - 데이터패칭(`useSavingsProducts.ts`), 필터링로직(`filteredSavingsProducts.ts`)등 구체적인 로직들은 훅을 통해 사용부에서 숨길 수 있도록 했습니다.
        2. 유틸함수
            - 금액 포맷팅(`formatCurrency.ts`), 숫자만 추출하는 함수(`extractNumbers.ts`) 등 재사용 가능한 순수함수들을 추상화 했습니다.
        3. 메인 컴포넌트 단순화
            - 메인 페이지 컴포넌트인 `SavingsCalculatorPage.ts`는 UI 구조와 "무엇을 하는지"만 보이고 구체적인 로직들은 숨기도록 했습니다.
- 확장 가능성: 기존의 코드를 건들이지 않고 새로운 요구사항을 추가할 수 있는 성질을 가진 코드라고 생각합니다. 예를 들어, 새로운 필터링 조건이 추가되어도 기존 컴포넌트는 수정하지 않고 훅 내부 로직만 확장하면 되도록 구조화하는 것이 중요합니다.
    - 현재 코드에서 확장 가능성에 신경 쓴 부분
        - 변경에 용이하게 잘 작성한 코드가 있는지 잘 모르겠습니다..
## 더 개선 하고싶은 부분
- 비즈니스 로직 코드가 기본적으로 복잡함
    - 유지보수 할 때 한번에 알아볼 수 있을까?
- 필터링 조건, 매직넘버, 옵션값, 정렬기준 등이 하드코딩 되어있음
    - 계산 결과공식이 여러곳에서 사용되거나 변경이 되었을 때 수정하기 불편함
- 상태관리
- 로딩, 에러처리등 사용자 관점에서 미흡한 부분

## 회고
기능 구현에 집중하기 보다 과제의 본질이었던 추상화와 확장가능성에 초점을 두지 못한 것 같습니다. 물론 시간이 더 있었더라도 완벽한 생각은 못했을것 같아요.
코딩을 하다보면 로직이 엉키게 돼서 다 뒤집어 엎고 새로 짜야할것만 같은 충동을 자주 느끼는데 이번에도 그랬습니다. 처음부터 설계를 잘 해놓으면 좋지만 그렇게 하기 어렵다는걸 알기에 매번 과제나 업무에서 생각을 먼저 하기 보다는 손부터 나갔던것 같아요. 이부분을 다들 어떻게 해결하는지 제일 궁금합니다.
